# Claude 冠军策略实盘交易系统实施方案

> 基线提交：`ffdbe65`
> 本地开发分支：`feature/live-claude-execution`
> 策略：`claude_board_wf_1m`
> 当前阶段：服务器主网微量验证运行中；仓库默认真实订单仍关闭
> 当前生效风控：20% 保证金、5 倍杠杆、最多 3 仓、21U 单笔名义上限
> 说明：前文保留了早期 10%/10x 本地实验记录，当前部署一律以 17.6 为准。

## 1. 目标与边界

在现有 Binance Hunter 项目中增加一套独立的真实执行系统，完整复刻 Claude 冠军策略的信号语义，并负责：

- 读取真实账户、余额、持仓、订单、成交、手续费、资金费率和强平价。
- 把策略信号转换为唯一、幂等的交易意图。
- 执行开空、交易所托管止损、动态追踪止盈和主动平仓。
- 处理部分成交、未知状态、断线、重启恢复、人工仓位和订单竞态。
- 对市价、可成交 IOC 限价和 Maker-first 做真实小额对照。
- 使用独立实盘账本，不污染现有三个纸面账户。

首版只交易 USD-M USDT 永续空单；不自动提现、不自动扩大资金、不同时复制三个纸面账户仓位。

## 2. 隔离结构

现有纸面生产服务保持不变。实盘使用独立包、独立进程和独立数据库：

```text
Claude Board 1m 已收线信号
        |
        v
TradeIntent（唯一 intent_id）
        |
        v
账户级风控 -> 执行方式路由 -> Binance OMS
        |                         |
        |                         +-> PAPI 签名 REST 普通订单
        |                         +-> PAPI Algo 条件单
        |                         +-> Portfolio Margin 私有用户数据流
        v
SQLite 实盘账本 + 对账 + SAFE_HALT
```

运行模式：

| 模式 | 行情/账户 | 发送订单 |
| --- | --- | --- |
| `paper` | 本地 | 否 |
| `dry_run` | 真实行情和真实账户 | 否 |
| `testnet` | 仅标准 USD-M 账户测试网；Portfolio Margin 不支持 | 是，需授权 |
| `live_micro` | 主网微量 | 是，需授权 |
| `live` | 主网正式 | 是，需授权 |

任何真实模式都必须同时满足：

1. `live_trading.enabled=true`。
2. `live_trading.real_order_enabled=true`。
3. 命令包含 `--confirm-real-orders`。
4. 命令显式给出 `--max-notional-usdt`。
5. 命令包含未过期、未使用的一次性 nonce。

## 3. 密钥安全

只从进程环境读取：

```text
BINANCE_API_KEY
BINANCE_API_SECRET
```

安全要求：

- 不读取项目内明文配置，不把用户密钥文件复制进仓库。
- 不在日志、异常、SQLite、前端、报告或通知中保存密钥和签名。
- API Key 只开放合约交易，关闭提现权限，建议绑定 IP 和独立子账户。
- 本地主网验证结束后更换一次密钥，再部署服务器。
- 交易进程永远不持有提现权限；外部提现保持人工操作。

## 4. 策略一致性

实盘信号继续由 `BoardWaterfallEngine` 产生：

- 只处理 Binance `x=true` 的已收线 `1m` K 线。
- 24h 涨幅不低于 40%。
- 当前收盘价从最近 60m 高点回落不低于 7%。
- 最近 60m 成交额不低于 300,000 USDT。
- 同币冷却 20 分钟，单币单日最多 8 次。
- BookDepth 仍按当前冠军策略配置做标记或过滤。

纸面引擎负责判断策略生命周期；真实仓位只以 Binance 成交和持仓为准。策略发出信号不等于已经成交。

## 5. 订单状态机与幂等

```text
CREATED
-> RISK_APPROVED / RISK_REJECTED
-> SUBMITTING
-> ACKED / PARTIALLY_FILLED / FILLED
-> PROTECTED
-> CLOSING
-> CLOSED / CANCELLED / EXPIRED / EXCHANGE_REJECTED / UNKNOWN
```

规则：

- 每个信号生成确定且唯一的 `intent_id` 和 `clientOrderId`。
- 同一 `intent_id` 重放不会再次下单。
- 部分成交按“累计成交量减已计入成交量”更新仓位，防止重复累计。
- 已成交但进程在写仓位前崩溃，重启后查询订单并只补记未应用部分。
- 超时、5xx 或连接中断造成执行状态未知时，先查订单和成交，绝不盲目重发。
- 退出只减少真实剩余数量；IOC/Maker 部分平仓后，对剩余数量降级为市价平仓。

## 6. 开仓执行方式实验

策略触发点处于快速下跌阶段，首选目标不是最低手续费，而是最高净执行价值：

```text
净执行价值 = 成交后的策略收益 - 手续费 - 滑点 - 漏单机会成本
```

本地真实微量阶段对照三种方式：

### 6.1 市价开空

- `MARKET SELL`。
- 成交确定性最高，适合价格快速逃逸、bid 深度塌陷的瀑布。
- 记录真实 taker 手续费、成交延迟和滑点。

### 6.2 可成交 IOC 限价

- 以买一价减允许滑点形成最低可接受卖价。
- 立即成交可成交部分，剩余自动取消。
- 限制极端滑点，但可能只成交部分或完全漏单。

### 6.3 Maker-first

- 在卖一价挂 `GTX/Post Only SELL`。
- 只等待 150/250/350/500ms 中的实验值。
- 查询累计成交量，撤销未成交部分，再对剩余数量降级市价。
- 新增成交只应用一次，部分仓位立即获得交易所保护。

每个真实策略信号只随机分配一种真实执行方式，不能把一个信号下成三笔仓位。三种方式按时间外样本比较：

- 首次/完全成交率。
- P50/P90/P95 成交延迟。
- 相对信号价和触发时盘口的滑点。
- maker/taker 手续费。
- 部分成交与撤单竞态。
- 漏单后 5s/30s/1m 的机会成本。
- 成交后短期 MFE/MAE 和整笔净收益。

## 7. 风险定仓

本地主网冠军账户使用与纸面 `claude_drawdown10` 相同的资金控制：

- Portfolio Margin 统一账户、双向持仓模式，空头订单固定使用 `positionSide=SHORT`。
- 交易所杠杆 10x。
- 同时最多 3 个真实仓位。
- 基础保证金比例为策略实现权益的 10%。
- 实现权益回撤 `<5% / <10% / <15% / >=15%` 时，仓位系数依次为 `1.0 / 0.75 / 0.50 / 0.25`。
- 绝对名义金额由命令显式限制。

计算：

```text
strategy_realized_equity = initial_equity
                           + REALIZED_PNL
                           + COMMISSION
                           + FUNDING_FEE
                           + INSURANCE_CLEAR

drawdown = 1 - strategy_realized_equity / strategy_peak_equity
target_margin = strategy_realized_equity * 10% * drawdown_factor
target_notional = target_margin * 10x

final_notional = min(
  target_notional,
  exchange_available_margin * leverage,
  absolute_notional_cap,
  允许滑点范围内盘口容量的 20%
)
```

策略初始权益、历史峰值、当前回撤和系数持久化在实盘账本。现金转入、转出和提现不计入策略盈亏，既不会虚增复利基数，也不会错误触发缩仓；真实可用保证金仍是最终硬上限。

数量和价格全部用 `Decimal`，按 `exchangeInfo` 的 `tickSize`、`stepSize`、`MARKET_LOT_SIZE`、`minQty` 和 `MIN_NOTIONAL` 对齐。最小订单超出风险上限时跳过，不为凑最小金额放大风险。

开仓前拒绝条件包括：

- 非 USDT 永续或非 `TRADING`。
- 非允许币种。
- 已有同币仓位或达到最大持仓数。
- 盘口价已从信号价逃逸超过上限。
- 止损无效、余额不足、深度不足。
- 非 Portfolio Margin 风险仓模式下近似强平距离不足。

Portfolio Margin 的名义仓位约等于策略实现权益，而不是 10 倍账户权益，因此不使用按交易所设置杠杆反推的简化强平距离。成交后始终读取交易所实际强平价；强平价未比结构止损高出足够缓冲时立即市价平仓并熔断。

## 8. 结构止损

第一次真实成交后立即通过 Binance Algo Service 创建：

```text
STOP_MARKET BUY
quantity=当前空仓数量
positionSide=SHORT
workingType=CONTRACT_PRICE
priceProtect=false
```

止损价：

```text
max(
  60m 低点后反弹高点 * 1.01,
  实际平均成交价 * 1.015
)
```

- 保护单必须在配置时间内确认有效。
- 初次保护失败立即 `MARKET BUY reduceOnly` 平仓并进入 `SAFE_HALT`。
- Maker 部分成交后新增成交会重新计算均价和止损。
- 结构止损更新时先确认新单，再撤旧单；新单失败则保留旧保护。
- 任何时刻真实仓位都不能依赖本地进程作为唯一止损。

## 9. 动态止盈与超时退出

沿用冠军策略：

- 浮盈达到 3.5% 后激活追踪。
- 从最低价反弹 3% 形成追踪退出价。
- 使用上一根已收线信息，避免同根未来函数。
- 最近 10 根已收线主动卖盘均值仍不低于 48% 时，认为卖方仍占优，暂不部署追踪退出。
- 最长持有 240 分钟。

实盘在每根 1m 收线后计算“下一分钟保护状态”：

1. 卖压持续时撤销或不部署追踪单，结构止损继续存在。
2. 卖压减弱时创建新的追踪 `STOP_MARKET`。
3. 新追踪单确认后再撤旧追踪单。
4. 如果挂单前盘口已经越过追踪价，直接市价平仓，避免挂一个已失效的条件单。

硬止损退出始终市价。非紧急止盈/超时退出也参与市价、IOC、Maker-first 对照，但等待时间更短，且未成交余量必须降级处理。

## 10. WebSocket 与 REST

独立维护：

- 公共行情 WebSocket：全扫描池已收线 1m。
- 交易热路径：Portfolio Margin 使用 PAPI 签名 REST 普通订单和 PAPI Algo API；执行请求绝不自动重试。
- 私有用户流：`TRADE_LITE`、`ORDER_TRADE_UPDATE`、`ACCOUNT_UPDATE`、`ALGO_UPDATE`。

REST 只用于：启动预检、`exchangeInfo`、信号时盘口/深度、未知订单查询、断线对账、余额/持仓/收入核对。标准 USD-M 可使用交易 WebSocket；Portfolio Margin 当前订单热路径使用官方 PAPI 签名 REST，执行请求不自动重试，未知状态必须查询确认。

私有流在全市场 K 线预热前启动。断线时：

- 立即禁止新开仓。
- 已有仓位继续受交易所止损保护。
- 重连后查询账户、仓位、普通订单、Algo 订单和收入。
- 完整对账通过后，才允许清除由私有流断线造成的熔断。

## 11. 重启与对账

启动顺序：

1. 打开实盘 SQLite 账本。
2. 校准 Binance 时间。
3. 读取账户、单/双向模式、真实持仓、普通挂单和 Algo 挂单。
4. 恢复本地未完成订单，并补应用已成交但未入账的增量。
5. 核对本地与交易所仓位。
6. 确认每个仓位有结构保护。
7. 启动私有流和周期对账。
8. 预热策略 K 线并启动公共行情流。

异常处理：

- 交易所有本地未知仓位：`SAFE_HALT`，不自动接管人工仓位。
- 本地有仓位而交易所已空：视为 Algo 或外部减仓完成，记录真实成交后关闭本地仓位，不重新开仓。
- 数量不一致：熔断。
- 结构保护丢失：先补保护；失败则紧急平仓。
- 未知订单无法查询：熔断，不重发。

策略事件、私有流事件、追踪单更新和周期对账使用同一异步执行锁，避免同一时刻重复平仓或错算仓位。

## 12. 账本与收益

独立数据库默认：`backend/storage/live_trading.db`。记录：

- 策略意图和数据截止时间。
- 普通订单、Algo 订单和所有状态变化。
- 每次真实成交、成交角色、手续费和已实现盈亏。
- 真实持仓、均价、数量、止损、追踪价和强平价。
- 账户余额快照、资金费率和 Binance income。
- 熔断、重连、对账和通知结果。

每日亏损熔断只统计：

```text
REALIZED_PNL + COMMISSION + FUNDING_FEE + INSURANCE_CLEAR
```

`TRANSFER` 不计入策略盈亏，因此转入/转出不会被误判成交易回撤。首版关闭自动转出。以后只有空仓、无挂单、对账完成并保留保证金储备时，才考虑把历史高点之上的新增已实现利润按固定比例内部划转到现货；外部提现仍人工执行。

## 13. 前端与通知

后端提供 `/api/live/summary`，只包含脱敏账户快照、真实仓位、订单、成交和熔断状态。默认 `dashboard_enabled=false`，防止 Vercel 公网页面泄露真实余额。

启用后瀑布页显示：

- `dry_run/testnet/live_micro/live` 模式。
- 余额、可用保证金和真实持仓数。
- 实际均价、数量、结构止损、追踪价、强平价和保护状态。
- 真实订单执行方式与状态。
- 信号到提交/首笔成交/完全成交延迟及相对信号价、到达盘口的滑点。
- 10% 回撤梯当前实现权益、回撤和仓位系数。
- `SAFE_HALT` 原因。

企业微信只发送重要事件：真实开仓/平仓结果、保护状态、拒单和熔断。`dry_run` 默认不推送。通知不包含 API Key、签名和内部请求参数。


```powershell
python run.py live-run --mode live_micro --max-notional-usdt 10 `
```

## 15. 本地验证顺序

1. 离线测试订单状态机、部分成交、重复信号、未知状态、保护失败和重启恢复。
2. 使用真实密钥执行 `live-preflight`，只读账户，不下单。
3. `dry_run` 运行真实行情和账户，验证信号、风控和账本。
4. 标准 USD-M 可使用测试网；Portfolio Margin 没有等价测试网，直接从主网远价挂撤单和最小名义金额开始。
5. 主网远价 Post Only 挂单并撤销。
6. 主网最小金额分别测试 `market`、`ioc`、`maker_first` 开平仓。
7. 主网微量策略运行 7～14 天且至少 30 笔已平仓交易。
8. 比较真实滑点、手续费、成交率、漏单机会成本和纸面信号。
9. 用户再次确认后才推送和部署服务器。

## 16. 验收标准

- 没有重复订单或重复应用成交。
- 重启后订单、仓位和保护单完整恢复。
- 每个真实仓位都有交易所托管结构止损。
- 保护失败时在 1 秒目标内紧急平仓。
- 退出后无残余真实仓位或遗留 Algo。
- 账户、成交、手续费、资金费率和本地账本可对账。
- 纸面与实盘信号决策时间一致，差异只来自风控和执行。
- 私有流断线、状态未知、限流和进程崩溃不会产生重复开仓或裸仓。
- 市价、IOC、Maker-first 的结论同时考虑成交率和漏单成本，不能只看手续费。
- 实盘分支可以在用户明确批准后推送；密钥、本地运行配置和账本不得推送。服务器部署和扩大资金仍需单独批准。

### 17.3 2026-07-18 最终执行审查与成交质量记录

每个真实普通订单新增以下持久化字段，进程重启后仍可审计：

- `submit_time`：请求实际发送前的本机时间。
- `ack_time`：本机收到交易所 ACK/查询确认的时间。
- `first_fill_time`：交易所首笔成交时间。
- `final_fill_time`：交易所完全成交时间。
- `signal_to_submit_ms`、`submit_to_ack_ms`、`submit_to_first_fill_ms`、`signal_to_fill_ms`（信号到首笔成交）、`signal_to_final_fill_ms`（信号到完全成交）。
- `slippage_bps`：相对纸面信号成交基准的总滑点。
- `arrival_slippage_bps`：相对 OMS 收到信号后买一/卖一可成交价的订单执行滑点。

两种滑点都统一为：正数表示不利成交，负数表示价格改善。总滑点用于判断实盘是否偏离纸面假设；到达滑点用于公平比较 `MARKET`、`IOC` 和 `maker_first`。`/api/live/summary` 与前端真实订单卡片展示信号到完全成交延迟及两种滑点；`dashboard_enabled=false` 仍是默认值。

信号和成交时间来自 Binance；提交与 ACK 时间使用启动/周期对账得到的 Binance 时间偏移校正，四段延迟在同一交易所时钟域计算，避免本机时钟误差污染毫秒级统计。

本轮审查修复了以下真实资金边界问题：

- 加仓、追加部分成交或部分平仓后，即使止损价格没有变化，也强制按交易所真实剩余数量替换结构止损和追踪单；Algo client ID 包含数量，避免重复 ID。
- 重启/周期对账不再只检查 Algo ID 是否存在，还核对交易所仓位数量与保护数量；不一致时以交易所仓位为准、先补新保护、再撤旧保护，并进入 `SAFE_HALT`。
- 取消追踪单必须由撤单响应或查询确认终态后才清除本地状态；无法确认时保留保护状态并熔断，防止交易所遗留条件单意外平仓。
- 新追踪保护单状态无法确认时保留旧追踪保护，不让异常退出行情主循环，同时进入 `SAFE_HALT` 等待对账。
- 替换保护单时如果旧单无法确认撤销，保留新保护并熔断；如果数量变化后新保护无法建立，立即市价平掉剩余仓位。
- 正常平仓后的结构止损和追踪单也必须由撤单响应或查询证明终态；遗留状态无法确认时立即熔断，防止旧条件单影响下一次同币持仓。
- 条件单令交易所仓位归零时，同一次对账立即撤销仍在交易所挂着的同仓位结构止损/追踪单，不等待下一轮对账。
- 实际开仓成交相对信号价或到达盘口的不利滑点超过 `max_entry_slippage_bps` 时，立即市价平仓并进入 `SAFE_HALT`；滑点上限不再只是记录指标。
- 主动退出、兜底市价退出或紧急退出被交易所明确拒绝时，订单会落为 `EXCHANGE_REJECTED`，剩余仓位立即恢复为 `open` 并熔断新增风险，不会停在假的 `closing` 状态阻塞后续保护。
- 开仓前设置杠杆/保证金模式失败时明确记录为“前置配置失败”，订单不会进入在途恢复队列，更不会在重启后被误判为已发送但状态未知。
- `SAFE_HALT` 保存所有不同原因并持久化。只有私有流或关键 REST 对账重新连接，并完成账户、持仓、普通订单、Algo 条件单的权威一致性核对后，系统才自动清除对应的连接类熔断；保证金、日亏损、未知订单、仓位或保护不一致等风险熔断不会自动解除。
- 私有用户数据流使用 `private_stream_stale_seconds` 控制心跳与失联探测；当前默认 10 秒，连接异常时先熔断新增仓位，重连并完成交易所对账后才处理恢复。
- 收到 `listenKeyExpired` 时必须强制关闭旧私有流、申请新 listenKey 并完整对账；成功后只清除 `listenkeyexpired/private_stream_failed` 这类已恢复原因，保证金、未知订单、仓位或保护不一致等熔断继续保留。进程重启时也执行同样的连接与对账证明，避免已恢复的过期 listenKey 永久阻止后续信号。
- 周期对账改为两层：账户、持仓、普通订单、Algo 条件单构成关键快照，连续三轮无法确认才熔断；时间同步和收入历史是可延后任务，单独重试、记录端点故障，但不会因为一次辅助接口抖动阻断开仓。收入同步失败期间沿用已确认账本，恢复后自动补账。
- 只读 REST 使用线程隔离的持久 HTTPS 连接池；同一轮关键快照顺序复用连接，避免六个接口同时握手。GET 请求使用四次指数退避和随机抖动，错误日志包含具体端点；交易写请求仍然只发送一次，未知执行状态必须查询确认，绝不自动重发。
- 网络对账类熔断在后续一次完整关键对账通过后自动解除，并发送企业微信恢复通知及 `CONNECTIVITY_AUTO_RECOVERED` 审计事件，不再需要人工修改数据库或重启进程。
- Portfolio Margin 的 `riskLevelChange`（`MARGIN_CALL`、`REDUCE_ONLY`、`FORCE_LIQUIDATION`）会立即熔断新增仓位、发送风险通知并触发交易所对账，不再只识别旧式 `MARGIN_CALL` 事件名。
- 同一根 1m K 线先平空、再重新开空时，实盘成功必须按 `position_id` 精确匹配。若平仓失败，旧仓位不会被误认成新开仓，纸面提前计入的退出盈亏会回滚。
- 订单 ACK 不再被当作成交；Portfolio Margin 返回缺少均价时继续查询订单，并在必要时从 `userTrades` 恢复 VWAP 和真实成交时间。
- 条件止损先令交易所仓位归零、私有成交事件稍后到达时，对账会主动从 `userTrades` 恢复真实退出价和盈亏后再关闭本地仓位；恢复失败会持久化熔断，不会把退出伪记为零价零收益。

策略一致性边界：

- 信号完全由同一个 `BoardWaterfallEngine` 产生，只使用 Binance `x=true` 的已收线 1m 数据；入场标签、20 分钟同币冷却、结构止损、3.5% 激活/3% 回弹追踪、10 根主动卖盘延续门和 240 分钟超时逻辑与纸面策略共用代码。
- 交易所结构止损会在分钟内触发，比纸面回放等到 1m 收线检查更快。这是实盘保护层的有意加强，不是信号模型变化。
- 实盘仓位控制已切换为纸面 `claude_drawdown10` 的 10% 保证金、10 倍杠杆和实现权益回撤梯；实盘额外限制最多 3 仓、绝对名义金额和盘口容量，因此并发受限时可能少做信号，但单笔仓位规则一致。
- BookDepth 当前按冠军策略配置做增强标记，`bookdepth_filter_mode=false` 时缺失或未确认不会阻止信号；该行为与现行纸面冠军策略一致。

验证结果更新以后续 17.4 为准。默认配置仍禁止真实订单。

结论：正常行情、正常网络和已覆盖故障下，系统可自主完成信号、风控、开仓、成交确认、交易所结构止损、动态追踪止盈、主动/超时平仓、对账和重启恢复；遇到未知执行状态、保护单无法确认、外部仓位/订单或状态不一致时会停止新增风险并保留减仓能力。它不会在严重不确定状态下“自动猜测后继续交易”。正式扩大资金前仍必须完成本地主网微量连续 7～14 天、至少 30 笔闭环交易，并逐笔核对延迟、滑点、手续费、保护单和重启恢复。

### 17.5 2026-07-23 自主监管与连接熔断恢复

本地主网连续运行暴露了 Binance Portfolio Margin REST 偶发 TLS EOF 的问题。旧实现每 30 秒并发请求六个接口，任一辅助接口失败就判定整轮对账失败，连续三轮后进入 `SAFE_HALT`；即使交易所账户、持仓和保护单实际正常，也需要人工清理熔断。现已按交易安全等级重构：

### 17.6 2026-07-24 服务器同源信号与无人值守执行

本地连续运行证明，独立实盘进程再订阅约 400 个币的公共 1m WebSocket
会产生两个无法接受的问题：断网后临时 Windows 进程可能直接消失；任一
连接漏掉一根 K 线，就会让要求连续 1,441 根数据的 Claude 标签与服务器
纸面信号分叉。因此服务器版改为单一策略数据源：

```text
服务器公共行情 WebSocket
  -> 唯一 BoardWaterfallEngine
  -> hunter.db 中有序 signal outbox + protection state
  -> 实盘进程 100ms 只读游标
  -> OMS / Binance 私有流 / 交易所保护单
```

关键约束：

- 纸面监控是唯一 K 线、选币和策略计算进程。实盘不再启动第二套公共行情、
  预热数据库或 `BoardWaterfallEngine`。
- 信号先写入带 `AUTOINCREMENT` 序号的 outbox，再更新三个纸面账户和发送
  企业微信；推送超时不再增加真实下单延迟。
- 实盘游标持久化在独立 `live_trading.db`。首次部署从 outbox 尾部开始，
  不执行历史信号；重启后按序续读，确定性 `intent_id` 再做一层去重。
- 新开仓信号超过 30 秒自动作废。纸面监控健康或最新已收线 1m 数据超过
  150 秒时暂停新开仓，恢复后自动继续；退出信号仍允许执行。
- 纸面引擎把 `trail_price`、`arm_trail` 和主动卖盘延续判断写入保护状态表。
  实盘只镜像交易所 Algo 保护，不重新计算退出策略。
- 市场快照读取失败发生在订单进入 OMS 之前，只退避重试，不写永久熔断。
  私有流断开、关键 REST 连续失败会暂停开仓，重连并完成权威对账后自动恢复。
- 共享 SQLite 短暂锁定或不可读时在进程内指数退避并重建只读连接，不停止
  私有用户流和周期对账；源异常期间丢弃新开仓，但继续按序执行所有退出信号。
- 保护价已越过时直接复用该次盘口快照发出退出，不重复请求 REST，不增加
  追踪止盈的竞态窗口。
- 结构止损缺失、保护数量不符或撤单状态未知，只有后续交易所快照能证明
  新保护已存在或旧单已消失时才自动解除。未知成交、外部仓位/订单、保证金
  风险与日亏损熔断不会自动猜测恢复。
- systemd 使用 `Restart=always`，每次启动由 root 服务包装器生成并消费新的
  一次性订单授权 nonce。仍同时要求私密配置启用真实订单、明确
  `--confirm-real-orders` 和 21U 绝对名义上限。
